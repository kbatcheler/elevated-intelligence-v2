- Phase K gate met (milestone): Tier 3 lands per-tenant cryptographic isolation, no standing human access, and a hash-chained provenance ledger, backend only (the portal security, break-glass, connections, and provenance screens are Phase L). Each signal value is sealed with a fresh AES-256-GCM data key (DEK) wrapped by a per-tenant 32-byte key (KEK) held in a local KMS, stored as an envelope `{v,alg,keyRef,iv,tag,ct,wrappedDek}` inside the existing `derived_signals.value` jsonb (no column churn); the derived-set root hash is still over the plaintext math. A swappable cloud/customer KMS adapter reports "available, not connected" until a customer key is configured, never a fabricated wrap. Crypto-shred is real: revoking a tenant key destroys the KEK material first and only then commits the revoked status and stamps `revokedAt` (a failed destroy throws before any status change, so a revoked key is never reported while its material survives), emits a structured revocation log line, and leaves the ciphertext inert; both reads require an active tenant key and verify each envelope's `keyRef` against it before decrypting, so both the human and machine reads then fail loud with a typed `crypto_shredded` error rather than any plaintext or empty grounding. No standing access: a human raw-signal read requires tenant access plus an active, unexpired, unrevoked break-glass grant for every role including owners, and every access appends an event to the new `access_grant_events` table; the in-boundary machine-grounding read is a separate service API, exempt by design, never a middleware bypass. The provenance ledger exposes only `appendEntry` and `verifyChain` (no update or delete), serializes per-tenant appends on a Postgres advisory lock, and links entries by `contentHash = sha256(canonical({tenantId, claimPath, sourceRef, prevHash}))` over source references only. outside_in is byte-for-byte unchanged (no key, no encryption, grounding regression still identical). Typecheck, build, and the full suite are green (123 tests across 15 files, up from 96; 27 new across crypto round-trip and tamper, key-lifecycle revoke failure-injection, ledger order and tamper, and the HTTP security surface); zero new npm deps (node:crypto, the pg already in the lockfile, global fetch, workspace packages); the long-dash sweep is zero across source and all 24 data tables (now including `kms_local_keys` and `access_grant_events`). Logged drift: the default KMS is a software key store on operator-controlled Postgres (`kms_local_keys`), not a hardware security module, and the customer-managed-key path is declared and stubbed honestly until configured; database-role append-only on the ledger table (revoking UPDATE and DELETE) is a deployment-time hardening left to the operator, with the hash chain plus serialized append as the application-level integrity control.
